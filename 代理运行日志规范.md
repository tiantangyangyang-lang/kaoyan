# 代理运行日志规范

> 目标：让每次 Claude Code / DeepSeek 任务都可追踪、可复核、可判定失败，而不是只留下聊天输出。

## 1. 每次运行必须生成的目录

```text
content/reports/agent-runs/{run-id}/
  run-manifest.json          # 启动参数、时间、任务、退出码和最终状态
  prompt.md                  # 本次实际发送给代理的完整任务提示
  source-mirror/             # 本批次供代理读取的来源副本
  terminal.log              # Claude Code / DeepSeek 完整终端输出
  source-before.json         # 来源库运行前快照
  source-after.json          # 来源库运行后快照
  source-integrity.json      # 来源库前后对比结果
  output-before.json         # 项目运行前文件快照
  output-after.json          # 项目运行后文件快照
  output-integrity.json      # 是否写入任务允许范围之外
  agent-result.json          # 代理按固定格式提交的结果
  agent-report.md            # 给人阅读的任务报告
  final-summary.md           # 启动脚本生成的最终验收摘要
```

日志目录只追加，不复用旧批次目录。`run-id` 使用时间戳和任务名生成。

## 2. 三层日志

### 第一层：终端事实

`terminal.log` 保存代理的完整 stdout/stderr，用于排查命令、错误和代理说明。

### 第二层：机器可读结果

代理必须生成 `agent-result.json`：

```json
{
  "schemaVersion": "agent-result-v1",
  "runId": "20260614-210000-cc-math1-2020",
  "task": "cc-math1-2020",
  "status": "completed_with_warnings",
  "summary": "完成数学一 2020 候选转换，存在 3 个需人工确认公式",
  "counts": {
    "inputFilesRead": 12,
    "itemsGenerated": 23,
    "itemsSkipped": 0,
    "warnings": 3,
    "errors": 0
  },
  "changedFiles": [],
  "createdFiles": [],
  "sourceFilesModified": [],
  "commandsRun": [],
  "checks": [],
  "warnings": [],
  "errors": [],
  "humanReviewRequired": [],
  "nextRecommendedTask": null
}
```

允许的 `status`：

- `completed`
- `completed_with_warnings`
- `blocked`
- `failed`

不得使用“差不多完成”“应该没问题”等模糊状态。

### 第三层：人工可读报告

`agent-report.md` 必须回答：

```text
结论：
读取了什么：
生成了什么：
修改了什么：
没有修改什么：
运行了哪些检查：
失败与警告：
需要人工确认：
下一批是否可以开始：
```

## 3. 自动失败条件

启动脚本遇到以下任一情况，最终状态必须为失败：

- Claude Code / DeepSeek 进程退出码非 0。
- 任一来源库前后快照不同。
- 代理写入当前任务允许目录之外。
- 缺少 `agent-result.json`。
- `agent-result.json` 不是合法 JSON。
- `agent-result.json` 的 `runId` 或 `task` 与当前运行不一致。
- 代理报告来源文件被修改。
- 代理将内容标记为 `approved` 或 `published`。
- 输出写入当前任务允许范围之外。
- `agent-result.json` 报告错误数量大于 0。
- `agent-result.json` 中任一检查状态为 `failed`。

如果代理任务本身被阻塞，但来源库未变化，可以记录为 `blocked`，不得伪装为成功。

Claude Code 插件自动生成的 `.claude/logs/**` 会被项目输出完整性检查忽略，因为它属于工具运行日志；`.claude/` 下其他配置和技能文件仍受越界写入检查保护。

## 4. 来源完整性快照

每次运行前后，对本批次允许读取的来源目录记录：

- 相对路径
- 文件长度
- 最后修改时间 UTC
- SHA-256
- Git tracked / modified / untracked 状态
- 仓库 HEAD commit 与 dirty 状态

前后任一差异都写入 `source-integrity.json`。差异不自动回滚，必须由人工判断；脚本只负责立即判定失败。

正式转换和复核任务不使用 Claude Code `--add-dir` 暴露原始仓库。启动脚本先复制当前年份所需文件到运行目录中的 `source-mirror/`，代理只读取副本；这是防止误写来源库的主要边界，前后快照是第二层检测。

## 5. 人工验收门槛

代理状态为 `completed` 也不等于内容可发布。

只有满足以下条件，才允许把当前批次从 `review/` 移到 `approved/`：

- `source-integrity.json` 显示来源未变化。
- `agent-result.json` 无未处理错误。
- 自动 Schema、KaTeX、图片和来源追踪检查通过。
- 人工逐题对照 PDF 完成审核。
- 版权与使用授权明确。

## 6. 推荐判断方式

运行后先看：

1. `final-summary.md`：判断任务成功、失败还是阻塞。
2. `agent-report.md`：了解代理做了什么。
3. `source-integrity.json`：确认来源库未变化。
4. `agent-result.json`：查看数量、检查和人工审核项。
5. `output-integrity.json`：确认代理没有越界修改其他项目文件。
6. `terminal.log`：只有需要排错时再查看完整日志。
